### Provenance Chip Color Map (Phase 2 — new)

Each persona field maps to a distinct M3 color role. The mapping is a closed set — only these 8 fields are emitted by the LLM per the tool schema. Unknown/unrecognized field names fall back to `surface-variant` with `on-surface-variant` text.

The chip visual treatment uses Vuetify `v-chip` with `variant="tonal"` — this renders a lightly tinted background using the named color's container tone, with the on-container text color. No custom CSS is required.

| Persona Field | `color` prop | Background (M3 tonal) | Text (M3 on-tonal) | Semantic rationale |
|---------------|-------------|----------------------|--------------------|--------------------|
| `goals` | `primary` | `#FDD6FF` (primary-container) | `#603768` (on-primary-container) | Goals drive the product forward — brand purple |
| `pain_points` | `error` | `#FFDAD6` (error-container) | `#93000A` (on-error-container) | Pain = friction signal — red family |
| `daily_workflow` | `secondary` | `#F3DBF2` (secondary-container) | `#524153` (on-secondary-container) | Workflow is the ambient context — muted secondary |
| `vocabulary` | `tertiary` | `#FFDAD5` (tertiary-container) | `#673B36` (on-tertiary-container) | Language/terminology — warm terracotta |
| `tech_comfort` | `info` | `#DCEEFF` (info tonal, Vuetify derives) | `#1E4D8C` (on-info, registered in main.ts) | Tech affinity — blue analytical signal |
| `pet_peeves` | `warning` | `#FFF0C8` (Vuetify warning tonal) | `#7A5800` (Vuetify on-warning) | Irritants — amber/caution |
| `review_lens` | `success` | `#C8EDD8` (success tonal, Vuetify derives) | `#2D6A4F` (on-success, registered in main.ts) | Evaluative lens — green judgment signal |
| `(fallback / unknown)` | `surface-variant` | `#ECDFE9` (surface-variant) | `#4D444C` (on-surface-variant) | Graceful degradation |

Implementation note: `warning` and `success` tonal containers are derived by Vuetify at runtime from the registered `success: '#2D6A4F'` and the Vuetify default `warning` role. The `info: '#1E4D8C'` is already registered in `main.ts`. No new color registrations are required — `warning` uses Vuetify's default amber.

**Chip invariants (from D-05, D-06, D-07):**
- Chips are always visible — no hover-to-reveal
- `sources: []` = chip row is not rendered at all (no empty chip row placeholder)
- A finding with N sources renders N chips in a horizontal flex row with `ga-1` (4px gap)
- Chip color reflects the field type, not the section color

## New UI Elements — Interaction Contract

### 1. Provenance Chip Row (SectionCard.vue)

**Location:** Immediately below the finding text `<p>` within each `.finding-row`, sharing the same left-indented column as the finding text (not full-width).

**Vuetify component:** `<v-chip>` with:
- `size="x-small"`
- `variant="tonal"`
- `:color="fieldColor"` (from the map above)
- `class="font-weight-medium"` (weight 500)
- No click handler, no ripple (`ripple="false"`)

**Label:** The field name as-is from `sources[]` — e.g., `pain_points`, `review_lens`. No transformation to title case. Rationale: matches the YAML field names exactly; keeps the provenance link transparent and copy-pasteable.

**Layout:**
```
[1]  Finding text goes here, wrapping to multiple
     lines as needed.
     [pain_points] [review_lens]         ← chip row, left-aligned
```

**Chip row element:**
- `<div class="d-flex flex-wrap ga-1 mt-1">` containing the chips
- `v-if="item.sources && item.sources.length > 0"` gates the entire div
- Rendered inside the same flex column as `finding-text`, not the `finding-num` column

**Accessibility:**
- Chips are decorative metadata. Add `aria-label="Sourced from: pain_points, review_lens"` to the chip row div for screen reader context.
- No keyboard focus on chips (they are not interactive).

---

### 2. Read-Only Persona Display (InputPane.vue)

**Confirmation only — no visual change required (D-09).**

The existing `v-select` with `:model-value="'General Manager'"` and `readonly` is the correct implementation. Phase 2 must NOT add:
- A "read-only" or "locked" label or badge
- A tooltip explaining why editing is disabled
- A link to `agent-personas`
- Any edit icon (even greyed out)

The anti-gaming guardrail is enforced by the absence of any authoring path. The field looks exactly like a normal read-only form field — no affordance draws attention to the restriction.

**Audit checklist (executor confirms):**
- [ ] No `v-tooltip` wrapping the persona `v-select`
- [ ] No `mdi-lock` or `mdi-pencil-off` icon in the field
- [ ] No `hint` or `persistent-hint` text referencing editing
- [ ] No click handler on the persona field

---

### 3. Gateway Error State (ResultsPane.vue)

**Trigger:** `error.value` is non-null in `useValidator.ts` after a failed validation run. Error types: timeout, auth failure (401/403), rate limit (429), unexpected (catch-all).

**Placement:** Replaces the results area (same slot as the four `SectionCard` components). The loading state transitions to the error state — no partial results are shown alongside the error.

**Component structure:**

```
<v-alert
  type="error"
  variant="tonal"
  rounded="lg"
  class="mt-4"
  :icon="errorIcon"
>
  <template #title>
    {{ errorHeading }}
  </template>
  {{ errorBody }}
  <template #append>
    <v-btn
      variant="text"
      color="error"
      size="small"
      @click="$emit('retry')"
    >
      Try again
    </v-btn>
  </template>
</v-alert>
```

**Error classification and copy (see Copywriting Contract below).**

**Retry behavior:** The "Try again" button emits a `retry` event. The parent clears `error.value` and re-runs the validation. No automatic retry — user-initiated only.

**Transition:** Error state fades in using the same `fade-in` keyframe already defined in ResultsPane.vue (`opacity: 0 → 1` over 0.6s). No new animation required.

### Error States

Three distinct error messages based on error type detected in `useValidator.ts`. The executor implements error classification by inspecting the error message/code from the gateway response before setting `error.value`.

| Error Type | Heading | Body | Icon |
|------------|---------|------|------|
| Timeout | "The gateway took too long" | "The validator didn't get a response in time. Check your connection and try again." | `mdi-timer-off-outline` |
| Auth failure (401 / 403) | "Access denied" | "Your gateway credentials may have expired. Contact your ACV admin if this continues." | `mdi-lock-outline` |
| Rate limit (429) | "Too many requests" | "The gateway is rate-limited right now. Wait a moment and try again." | `mdi-speedometer-slow` |
| Unexpected / catch-all | "Something went wrong" | "The validator hit an unexpected error. Try again — if it keeps happening, check the engine logs." | `mdi-alert-circle-outline` |

Retry CTA label (all types): **"Try again"** (verb only — terse, not blaming)

### `provenance.ts` contract

```typescript
// apps/web/src/utils/provenance.ts
export const FIELD_COLOR_MAP: Record<string, string> = {
  goals:          'primary',
  pain_points:    'error',
  daily_workflow: 'secondary',
  vocabulary:     'tertiary',
  tech_comfort:   'info',
  pet_peeves:     'warning',
  review_lens:    'success',
};

export function chipColor(field: string): string {
  return FIELD_COLOR_MAP[field] ?? 'surface-variant';
}
```

This is the single source of truth for chip color. SectionCard.vue imports `chipColor` and passes it to `v-chip :color`.
